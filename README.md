## 기본 과제

### 필수

- [x] 반복 유형 선택
  - 일정 생성 또는 수정 시 반복 유형을 선택할 수 있다.
  - 반복 유형은 다음과 같다: 매일, 매주, 매월, 매년
    - 만약, 윤년 29일에 또는 31일에 매월 또는 매년 반복일정을 설정한다면 어떻게 처리할까요? 다른 서비스를 참고해보시고 자유롭게 작성해보세요.
- [x] 반복 간격 설정
  - 각 반복 유형에 대해 간격을 설정할 수 있다.
  - 예: 2일마다, 3주마다, 2개월마다 등
- [x] 반복 일정 표시
  - 캘린더 뷰에서 반복 일정을 시각적으로 구분하여 표시한다.
    - 아이콘을 넣든 태그를 넣든 자유롭게 해보세요!
- [x] 반복 종료
  - 반복 종료 조건을 지정할 수 있다.
  - 옵션: 특정 날짜까지, 특정 횟수만큼, 또는 종료 없음 (예제 특성상, 2025-06-30까지)
- [x] 반복 일정 단일 삭제
  - 반복일정을 삭제하면 해당 일정만 삭제합니다.
- [x] 반복 일정 단일 수정
  - 반복일정을 수정하면 단일 일정으로 변경됩니다.
  - 반복일정 아이콘도 사라집니다.

- [ ] 월간 반복 옵션:
  - 매월 특정 날짜에 반복되도록 설정할 수 있다.
  - 매월 특정 순서의 요일에 반복되도록 설정할 수 있다.
- [x] 반복 일정 전체 수정 및 삭제
  - 반복 일정의 모든 일정을 수정할 수 있다.
  - 반복 일정의 모든 일정을 삭제할 수 있다.

## 심화 과제

- [x] 이 앱에 적합한 테스트 전략을 만들었나요?

### 각 팀원들의 테스트 전략은?

테스트 코드 작성에 앞서, 팀 내에서 사전 회의를 통해 공통된 테스트 전략을 합의하였습니다. 
이 과정에서 각 테스트 유형의 범위와 역할을 명확히 정의하고, 반복 일정 관련 요구사항을 고려하여 유닛 테스트, 통합 테스트, 그리고 E2E 테스트를 수행하기로 결정하였습니다.

### 합의된 테스트 전략과 그 이유는 무엇인가요?

1. 유틸리티 함수: TDD 기반의 유닛 테스트 작성

유틸리티 함수는 독립적이고 단일 기능을 수행하므로, TDD 방식을 적용하여 견고한 단위 테스트를 작성하기로 하였습니다.
이를 통해 코드의 안정성을 보장하고, 향후 리팩토링 시에도 신뢰할 수 있도록 합니다.

2. 반복 일정 기반 이벤트 CRUD: 기존 통합 테스트 내 포함

반복 일정과 관련된 이벤트의 생성, 조회, 수정, 삭제 기능을 통합 테스트에서 검증합니다.

3. 유저 시나리오 기반의 E2E 테스트 작성

주요 사용자 시나리오를 기반으로 E2E 테스트를 작성하여, 전체 애플리케이션이 기대한 대로 동작하는지 검증합니다.
데이터베이스와의 상호작용을 포함하여, 실제 서비스 환경에서의 동작을 확인하는 것이 목표입니다.
UI를 포함한 E2E 테스트를 수행함으로써, 프론트엔드와 백엔드 간의 상호작용이 정상적으로 이루어지는지를 확인하고, 실제 사용자 경험을 시뮬레이션합니다.

- 통합 테스트와 E2E 테스트 간의 역할 분리

통합 테스트에서 CRUD 기능의 동작을 검증하여 서비스의 주요 기능이 정상적으로 작동하는지를 확인합니다.
E2E 테스트를 통해 실제 서버 환경에서 UI 상에서도 주요 핵심 시나리오가 정상적으로 실행되는지를 추가로 검증합니다.
